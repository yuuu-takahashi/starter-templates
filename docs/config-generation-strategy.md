# 設定ファイルの生成・共通化方針

各テンプレートの設定ファイルを**すべて生成ファイル**にし、**共通ソース（shared / languages）から一元管理**するための方針です。

---

## 現状

### すでに生成されているもの（`yarn generate:configs` / `yarn generate:devcontainer`）

| 種別 | ソース | 出力先 |
|------|--------|--------|
| Prettier | `shared/prettier/.prettierrc.json` | 各テンプレート `.prettierrc.json` |
| EditorConfig | `shared/editorconfig/.editorconfig` | 各テンプレート `.editorconfig` |
| .gitignore | `shared/gitignore/.gitignore.<stack>` | 各テンプレート `.gitignore` |
| .rspec | `languages/ruby/.rspec.common` + テンプレート別 require | rails / rails-api / sinatra |
| .rubocop.yml | `languages/ruby/.rubocop_*.yml` のマージ | rails / rails-api / sinatra / ruby |
| ESLint (nodejs) | `languages/node/eslint.config.base.mjs` | templates/nodejs/eslint.config.mjs |
| CI workflows | `shared/workflows/*.yml` | 各テンプレート `.github/workflows/*.yml` |
| dependabot | `shared/workflows/dependabot.yml` | 各テンプレート `.github/dependabot.yml` |
| README | `scripts/template-readme-config.ts` | 各テンプレート `README.md` |
| .env.example / .env.test | `shared/env/*`（head + db.env） | rails-api, sinatra |
| devcontainer | `shared/devcontainer/defaults.json` + generate-devcontainer.ts | 各テンプレート `.devcontainer/*` |

### まだ手動維持されているもの（共通化の候補）

| テンプレート | ファイル | 共通化のイメージ |
|-------------|----------|------------------|
| react | `eslint.config.js`, `tsconfig.json`, `tsconfig.app.json`, `tsconfig.node.json` | `languages/node/` または `shared/` に React 用ベースを置き、生成 |
| nextjs | `vitest.config.ts`, `vitest.setup.ts`, `vitest-env.d.ts`, `tsconfig.json` 等 | Node/Next 用の共通 tsconfig・vitest を shared に置き、生成 |
| rails | `eslint.config.js`, `tsconfig.json` | Ruby+JS 用 ESLint/tsconfig を languages に置き、生成 |
| sinatra | `.env.development` | shared/env から .env.example と同様に生成（開発用として出力） |

---

## 共通化の考え方

1. **単一ソース**  
   編集するのは `shared/` または `languages/` のみ。`templates/<id>/` 配下の設定ファイルはすべてスクリプトの**出力**とみなす。

2. **レイヤー分け**  
   - **横断共通**: Prettier / EditorConfig / .gitignore など → `shared/`
   - **言語・スタック別**: ESLint / RuboCop / tsconfig / vitest など → `languages/node/`, `languages/ruby/`
   - **テンプレート固有**: README の文言・手順だけ → `scripts/template-readme-config.ts` のような「設定オブジェクト」で保持

3. **既存パターンの延長**  
   - **コピー**: そのまま写す（prettier, editorconfig, workflows の多く）
   - **マージ**: ベース + フラグメント（RuboCop, 今の .rspec）
   - **テンプレート置換**: プレースホルダー置換（shared/env の `{{TEMPLATE_ID}}`, `{{ENV}}` など）

4. **「何をどこに出すか」の一覧**  
   設定の種類 × テンプレートの対応を一箇所で管理すると運用しやすい（下記「マニフェスト案」参照）。

## 推奨する進め方

### Phase 1: すぐやれるもの

- **.env.development**  
  `shared/env` の既存ロジックで、`example` / `test` に加え `development` を出力する。  
  → rails-api / sinatra の `.env.development` を生成ファイルにし、shared と同期させる。

### Phase 2: ESLint / TypeScript / Vitest の共通化

- **languages/node/** に以下を追加し、生成対象にする。
  - `eslint.config.react.mjs`（または .ts）→ `templates/react/eslint.config.js`
  - `eslint.config.rails.mjs`（Rails 用 JS/TS 用）→ `templates/rails/eslint.config.js`
  - 必要なら `eslint.config.nextjs.mjs` → nextjs
- **tsconfig**  
  - `shared/tsconfig/` または `languages/node/tsconfig/` に  
    `tsconfig.base.json`, `tsconfig.react-app.json`, `tsconfig.react-node.json`, `tsconfig.nextjs.json` などを置き、  
    `generate-configs.ts` で各テンプレートにコピーまたはマージして出力。
- **Vitest**  
  - `shared/vitest/` または `languages/node/vitest.config.base.ts` を用意し、  
    nextjs / react 用の `vitest.config.ts`, `vitest.setup.ts` を生成。

### Phase 3: マニフェストで「生成一覧」を明文化（任意）

「どのテンプレートにどの設定をどのソースから出すか」を一つの設定オブジェクトで管理する方法です。

```ts
// scripts/template-config-manifest.ts（案）
export const CONFIG_MANIFEST = [
  { template: "react",  out: "eslint.config.js",  source: "languages/node/eslint.config.react.mjs" },
  { template: "rails", out: "eslint.config.js",  source: "languages/node/eslint.config.rails.mjs" },
  { template: "nextjs", out: "vitest.config.ts",   source: "shared/vitest/vitest.nextjs.ts" },
  // ...
];
```

- `generate-configs.ts` がこのマニフェストを読み、**ループで一括生成**する。
- 追加・削除はマニフェストの行の増減だけで済み、どの設定がどのテンプレートに効くかが一覧で分かる。

---

## 運用ルール

- テンプレート直下の設定ファイルは**編集しない**。変更は必ず `shared/` または `languages/` を編集し、`yarn generate:configs`（必要なら `yarn generate:devcontainer`）で反映する。
- 生成ファイルには先頭に「Generated by ... — edit ... instead.」のコメントを入れ、ソースの場所を明示する（既存と同様）。
- 新規テンプレート追加時は、マニフェスト（または generate-configs 内の対応表）にそのテンプレート用のエントリを追加する。

---

## まとめ

| やり方 | 効果 |
|--------|------|
| 設定をすべて生成にする | テンプレート間のぶれがなくなり、修正は共通ソース一括で済む |
| shared / languages にソースを集約 | 共通化が進み、重複が減る |
| コピー・マージ・テンプレート置換のパターンを使い分け | 既存の generate-configs の延長で無理なく拡張できる |
| マニフェスト（一覧）を用意する | 「何がどこから出るか」が分かりやすく、追加・削除が楽になる |

まずは **.env.development の生成** から入り、続けて ESLint / tsconfig / Vitest を `languages/node/` や `shared/` に寄せて生成するのがおすすめです。
